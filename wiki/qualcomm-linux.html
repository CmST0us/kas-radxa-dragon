Qualcomm Linux 官方文档要点

面向 kas-radxa-dragon 项目（Radxa Dragon Q6A / QCS6490，基线 Qualcomm Linux QLI.1.5-Ver.1.1，scarthgap）整理 · 最后更新：2026-06-06

本页是对 Qualcomm Linux 官方文档两大入口 Build Guide（编译指南） 与 Yocto Guide（Yocto 指南） 及其全部子页面的深入提炼，结论已与 qualcomm-linux/* 的 GitHub 仓库源码交叉核实。 目的：在用 kas 维护本项目时，随手查到「官方是怎么定义/构建/刷写的」，以及「本项目的 kas 等价做法」。

⚠ 两条硬性使用须知

① 抓取/查阅方式：docs.qualcomm.com/bundle/publicresource/topics/<id>/<page>.html 是 JavaScript 动态渲染的壳，机器抓取只得空白。要拿正文，请改用静态形式 docs.qualcomm.com/doc/<docID>/topic/<page>.html。各 docID 见 第 15 节。

② 版本漂移：docs.qualcomm.com 默认展示较新的 QLI 版本（部分页面已是 QLI 2.0：内核 6.18、纯 kas、 meta-qcom-releases），与本项目的 QLI.1.5（内核 6.6.90） 不同。 本页所有事实均已对齐 scarthgap / QLI.1.5；遇到 build_landing_page 之类指向 QLI 2.0 的内容已剔除。 写配置前永远以本仓库 kas-radxa-q6a.yml 与检出的 layer 源码为准。

真值来源（source of truth）

官方文档是「人类可读说明」，而 检出到 layers/ 的 layer 源码（machine conf、recipe、bbclass） 才是事实。本页写到的 commit/分支/变量名，落地前请到对应文件复核。相关锁定版本见 components/layers.md。

两种 BSP 变体 —— 全栈贯穿的核心开关

变量 QCOM_SELECTED_BSP 选择 BSP 变体，影响内核 recipe、图形栈、DTBO 等：

  • custom（默认）含高通增值/优化闭源件：Adreno GPU、custom 6.6 内核、DSP、DTBO 支持、WLAN qcacld 等。
  • base尽量用纯上游：Mesa 图形、上游 stable 内核；仅少量尚在 upstreaming 的高通件例外。

kas 等价：在 local_conf_header 设 QCOM_SELECTED_BSP，或固化在 machine inc 中，免去 setup-environment。

3 · meta-qcom-hwe —— machine 配置层级（QCS6490 核心）

本项目最相关一节。详见 machine 组件页。

3.1 配置三层 require 链

一台机器的配置由三层 require 串成，板级 conf 只需 require SoC inc：

arch/arch-armv8-2a.inc                 # ARM 架构 tune
  └─ qcom-base.inc                      # 跨机器通用：BSP 选择、provider、fstype、固件、引导
      └─ qcom-qcs6490.inc               # SoC 层：SOC_FAMILY、KERNEL_CMDLINE、SoC 包组
          └─ qcs6490-<board>.conf       # 板级：KERNEL_DEVICETREE、MACHINE_FEATURES
命名差异：qcom-base.inc vs qcom-armv8a.conf

官方「自定义 machine」文档把通用层写作 qcom-armv8a.conf（那是 upstream meta-qcom（yoctoproject）的老路径， 含 DragonBoard/RB5 设备树）。但 QLI 产品线的 meta-qcom-hwe（scarthgap）实际源码里通用层是 qcom-base.inc。 无论文档怎么称呼，稳定可照搬的指令是「板级 conf require conf/machine/include/qcom-qcs6490.inc」。 meta-radxa-dragon 的 qcs6490-radxa-dragon-q6a 即仿此层级、复用 qcom-qcs6490.inc。

qcom-base.inc 的关键机制

  • QCOM_SELECTED_BSP ?= "custom" → 驱动 MACHINEOVERRIDES =. "qcom-${QCOM_SELECTED_BSP}-bsp:"
  • SOC_FAMILY 加 qcom: 前缀，去连字符派生 SOC_ARCH，按需进 PACKAGE_EXTRA_ARCHS
  • 图形 provider：custom=qcom-adreno，base=mesa（virtual/egl、virtual/libgles…）
  • 内核 provider：PREFERRED_PROVIDER_virtual/kernel custom=linux-qcom-custom，base=linux-qcom-base
  • IMAGE_FSTYPES 限 ext4（4096B 块对齐），mask 掉 tar.gz；OTA 镜像走 OSTree
  • 引导/固件：firmware-qcom-bootbins + firmware-qcom-partconf；bootimg ARM64 基址 0x80000000、4KB page
  • 控制台 115200 ttyMSM0；EFI 用 systemd-boot + UKI（/EFI/Linux）

3.2 qcom-qcs6490.inc（SoC 层，QCS6490 专属）

  • DEFAULTTUNE = "armv8-2a-crypto"
  • SOC_FAMILY = "qcm6490"
  • KERNEL_CMDLINE_EXTRA 含 pcie_pme=nomsi、kernel.sched_pelt_multiplier=4、 rcupdate.rcu_expedited=1、rcu_nocbs=0-7 等；调试构建经 DBG_CMDLINE 追加 earlycon、page_owner、qcom_scm.download_mode（即 DEBUG_BUILD=1 才有 earlycon）
  • 依赖固件包组 packagegroup-firmware-qcm6490
  • custom 专属 RDEPENDS：WLAN 内核模块（qcacld / qca6490）、sensor 包、子系统 ramdump 工具、secure-DSP 示例、 remote-debug-agent、fastrpc、modemmanager、networkmanager、pd-mapper
改内核 cmdline 的位置

要改内核命令行，编辑 meta-qcom-hwe/conf/machine/include/qcom-<SoC>.inc 里的 KERNEL_CMDLINE_EXTRA。本项目 bring-up 期间就是在 kas local_conf_header 临时追加 earlycon 等以提升早期可见性（见 flashing 与 log.md）。

QCS6490 官方板级 conf（meta-qcom-hwe/conf/machine/）

QLI.1.5 提供四个 QCS6490 套件：qcm6490-idp.conf、qcs6490-rb3gen2-core-kit.conf、 qcs6490-rb3gen2-vision-kit.conf、qcs6490-rb3gen2-industrial-kit.conf。 以 core-kit 为例：

  • require conf/machine/include/qcom-qcs6490.inc
  • MACHINE_FEATURES = "usbhost usbgadget alsa wifi bluetooth"
  • KERNEL_DEVICETREE：base 用上游 qcom/qcs6490-rb3gen2.dtb；custom 用带 addon 后缀的 DTB（FP1/FP2/FP2.5/FP3 + HSP 变体）
  • DTBO 经 KERNEL_TECH_DTBOS 指定，provider 为 qcom-graphicsdevicetree、qcom-videodtb、 cameradtb、wlan-devicetree —— DTBO 仅 custom 变体支持

注册用户鉴权：~/.netrc 加 chipmaster2.qti.qualcomm.com 与 qpm-git.qualcomm.com 凭据， token 经 qsc-cli show-access-token。未注册用户路线不需 netrc/token，靠公开预编译固件，编译命令一致。

6.3 固件源码与 extras（注册用户）

固件以源码提供时，git clone 自 qpm-git.qualcomm.com/.../qualcomm-linux-spf-*（如 ...-qimpsdk.git），meta-qcom-extras 取自 ...hlos_oem_metadata.git； 运行 python build.py --imf 生成 QCM6490 固件 zip：

  • QCM6490_bootbinaries.zip（引导固件）· QCM6490_dspso.zip（DSP）· QCM6490_fw.zip（固件分割）

关键环境变量：EXTRALAYERS、CUST_ID（no-modem 用 213195）、FWZIP_PATH、 HEXAGON_ROOT、LLVM（需 14.0.4 与 16.0.7）、QCOM_SELECTED_BSP=custom。

与本项目刷写的关系

这三个 zip 正是 第 8 节 EDL 刷写消费的引导固件/DSP/固件产物；未注册用户直接用预编译包。 本项目 edl-ng 单次合并写全 LUN0–5 即覆盖这套固件分区。

7 · 用户定制 → kas 映射

本项目把这些都落进 meta-radxa-dragon 的 machine/distro 文件或 kas local_conf_header。

# layers/meta-qcom-distro/conf/distro/<name>.conf —— 复制 qcom-wayland.conf 为模板
DISTRO_NAME = "..."
DISTROOVERRIDES = "test-dist"
DISTRO_FEATURES:append = " cramfs"       # 用 :append 避免顺序问题
# 验证：bitbake -e | grep '^DISTRO_FEATURES='

7.3 自定义 image recipe

官方该专页未公开取到正文；标准 Yocto 做法：新建 recipes-core/images/<img>.bb， require qcom-multimedia-image.bb 或 inherit core-image， IMAGE_INSTALL:append = " pkg1 pkg2"；新 layer 用 bitbake-layers create-layer 后加入 bblayers.conf。 落地前建议读检出的 meta-qcom-distro 源码核实确切基础镜像名。

kas 等价

上述 QCOM_SELECTED_BSP、DISTRO_FEATURES:append、IMAGE_INSTALL:append 都可写进 kas 的 local_conf_header，machine/distro 文件则放进 meta-radxa-dragon， 从而完全不依赖 setup-environment。见 kas-configuration。

8 · 刷写（EDL / qdl / firehose）

本项目用 edl-ng（scripts/flash-edl.sh），详见 topics/flashing.md。

工具栈

  • QDL（Qualcomm Device Loader）：官方主用，先上传 firehose flash loader 再写镜像
  • PCAT：provisioning 工具，-PLUGIN SD 做 UFS provision / 刷 CDT（需 QSC CLI + QUD）
  • 第三方等价物（同说 firehose 协议）：edl(bkerler)、qdl-rs、edl-ng（本项目）

进入 EDL（9008）模式

  • UART：reboot edl · ADB：adb shell reboot edl · 手动：长按 F_DL
  • 验证：lsusb 出现 ID 05c6:9008 Qualcomm, Inc. ... (QDL mode)
  • 主机 udev：/etc/udev/rules.d/51-qcom-usb.rules 给 05c6:9008 设 MODE="0666" GROUP="plugdev"

firehose 三件套与命令

文件作用
prog_firehose_ddr.elffirehose flash loader（先上传，后续都靠它写）
rawprogram*.xml「哪个镜像 → 哪个分区/LUN/起始扇区」
patch*.xmlGPT 表回填/补丁（写 backup GPT、修正 last_lba）
gpt_main*.bin / gpt_backup*.bin主/备 GPT 镜像，由 Ptool / gen_partition.py 从 partitions.conf 生成
 
# 标准刷写
qdl prog_firehose_ddr.elf rawprogram*.xml patch*.xml
# UFS provision（一次性）
qdl --storage ufs prog_firehose_ddr.elf <Provision file>
# 成功标志：终端打印 "partition 1 is now bootable"
# 反向 dump GPT 并生成 rawprogram：edl --genxml

官方刷写顺序：① 更新 udev ② 进 EDL ③ Provision UFS（一次性，分 LUN）④ 刷 CDT（如 CORE_KIT_CDT.bin）⑤ qdl/PCAT 刷镜像。

写全引导固件 = 覆盖 boot LUN

要写全引导固件须覆盖含 boot 的 LUN（UFS：LUN1=Boot A、LUN2=Boot B 的 xbl_a/b、xbl_config_a/b， 以及含 aop 的 LUN4）。这与仓库 commit b4404b6「edl-ng 单次合并写全 LUN0–5（含引导固件）」一致。

9 · 分区与 LUN 布局

分区为 GPT，A/B 双槽，由 Ptool 从 partitions.conf 生成。UFS 分 LUN：

LUN用途代表分区 / 镜像
LUN0元数据 / EFIssd（8KB）；部分布局把 efi(vfat, type-guid C12A7328-…, ~512MB) 放此
LUN1Boot A（引导固件）xbl_a(3604KB ← xbl.elf)、xbl_config_a(512KB)
LUN2Boot B（备份）xbl_b、xbl_config_b（GUID 不同于 A 槽）
LUN4固件aop_a(512KB ← aop.mbn)
其余OSrootfs/system(system.img)、persist 等

eMMC 对比：引导固件集中在 LUN0，主分区表只定义一次。改分区表 → 改 partitions.conf 后重生成 GPT。

Persist 分区

  • 存跨重启与 OTA 不变的 BSP 数据（DRM、传感器寄存器、WiFi/BT/Camera 校准）
  • 挂载点 /var/persist（ext4），由 systemd var-persist.mount 挂 /dev/disk/by-partlabel/persist
  • resize-partition@persist.service（recipe qcom-resize-partitions.bb）启动时扩容到最大可用
  • 删单文件，别整盘 wipe；验证 mount | grep persist

10 · 启动链 / systemd-boot+UKI / Secure Boot

完整启动链

PBL(bootROM) → XBL_SEC(EL3) → XBL_Loader(DDR 训练)
  → TZ/QTEE(EL3) + QHEE/Hypervisor(EL2)
  → UEFI/EDK2 + ABL(EL1) → systemd-boot → Kernel/UKI(EL1)

OS 段：UEFI 挂 EFI 分区里的 efi.bin(VFAT) → 执行 bootaa64.efi(systemd-boot) → 解析 loader.conf → 加载内核。

EFI 分区（efi.bin，VFAT）内容

  • bootaa64.efi（systemd-boot 管理器）· /loader/loader.conf · /loader/entries/（Type1 启动项）
  • 内核镜像 / initrd；启用 OSTree 时位于 /ostree/poky-<sha256>/
  • /loader/keys/authkeys/（secure boot 的签名 Auth 文件）

Type1（BLS）vs Type2（UKI）

  • Type1：/loader/entries/*.conf 分别指 kernel + initrd + cmdline
  • Type2（UKI）：kernel+initrd+cmdline+os-release 合并为单个 PE/EFI（stub linuxx64.efi.stub）， 由 linux-qcom-uki.bb 生成、esp-qcom-image.bb 打包进 efi.bin。更安全：签一次覆盖全部启动实体。
本项目 UKI 细节（已核实，见 log.md）

本项目 UKI 的 ukify 不含 --dtb（DTB 由 UEFI 从 dtb_a 分区加载，不进 UKI）； root=/dev/disk/by-partlabel/system；UKI 含 initramfs-qcom-image。 qcom-multimedia-image 是 host 镜像，Linux 跑在 Gunyah EL2 之上的 primary VM。

Secure Boot

  • UEFI 密钥：PK(仅1，最后注册) · KEK(多) · DB(白名单) · DBX(黑名单)
  • 签名工具 signing_tool.py（依赖 OpenSSL、sbsign、mtools、Python3），config.ini 配 operation/image_type/file_path
  • EFI/UKI(PE) 用 sbsign --key … --cert … bootaa64.efi bootaa64.efi 内嵌签名（解包 efi.bin → 签 → 重打包）
  • DTB(非 PE) 用 OpenSSL CMS 分离签名生成 *.dtb.sig（校验时两文件都要）
  • 开启 secure boot 后，刷入的 efi.bin/UKI 必须先签名，否则 UEFI 拒绝加载

11 · OTA 升级（Capsule + OSTree）

固件走 UEFI Capsule，OS/rootfs 走 OSTree，A/B 双槽 + systemd-boot 启动计数回滚。

固件（Capsule）

mkdir /boot/EFI/UpdateCapsule
scp <fw.cap> user@ip:/boot/EFI/UpdateCapsule/
echo -e -n "\x4\x0\x0\x0\x0\x0\x0\x0" > data.hex
efivar -n 8be4df61-93ca-11d2-aa0d-00e098032b8c-OsIndications -f data.hex -w   # 触发 capsule
reboot
# 校验：/sys/firmware/efi/esrt/entries/entry0/{last_attempt_status(0=成功),last_attempt_version,fw_version}

OS（OSTree）

ostree admin status
scp -r <repo> ...:/tmp  &&  ostree pull-local /tmp/<repo> <branch>
ostree refs  &&  ostree admin deploy <branch>  &&  reboot
ostree admin status   # 校验

回滚：systemd-boot 用文件名计数标签（+3 初始 → 失败递减 +2-1…+0-3），归零回退上一有效配置； 成功启动后 systemd-bless-boot.service 清标签标记 good。

12 · init 脚本 / 二级 VM / SD 启动 / efivar / Docker

系统 init 脚本（systemd，meta-qcom-hwe 提供）

  • 存储：var-persist.mount、resize-partition@.service（启动时 e2fsck/mkfs/resize2fs 扩容）
  • 属性：property-vault.service、persist-property-vault.service（跑 set-persist-prop.sh）
  • 系统：android-tools-adbd.service、pd-mapper.service(管 protection domains)、rsyslog、logrotate

二级虚拟机（Secondary VM）

  • Hypervisor Gunyah(Type-1)，VMM 用 crosvm(Rust)，命令带 --hypervisor gunyah
  • 构建：host 装 clang-11，bblayers 加 meta-rust(Rust 1.73.0)，crosvm 进 packagegroup
  • 流程：bitbake qcom-console-image(host) → bitbake multiconfig:qcom-guestvm:qcom-guestvm-image(GuestVM) → 拷到设备 → crosvm … run --disable-sandbox --hypervisor gunyah --root <rootfs> <kernel>

从 SD 卡启动（冗余/回退）

  • 设备节点 /dev/mmcblk1：p1 = 512MB vfat ← efi.bin；p2 = 10GB ext4 ← system.img
  • 写入：dd if=/tmp/efi.bin of=/dev/mmcblk1p1 bs=4M / dd if=/tmp/system.img of=/dev/mmcblk1p2 bs=4M
  • 作用：UFS/eMMC 的 EFI 分区损坏或 systemd-boot 失败时从 SD 回退；引导固件(XBL)仍在 UFS/eMMC，SD 不刷引导固件

UEFI 变量（efivar）

efivar [-n <GUID>-<Name>] [-f file] [-p 打印] [-w 写]；典型用途即上面 OTA 的 OsIndications 写入。

Docker

qcom-multimedia-image 默认含 Docker（packagegroup-qcom-containers，依赖 meta-virtualization）。 GPU 透传例：docker run -it --rm --device=/dev/kgsl-3d0 <image>。首条 docker 命令后守护进程自启。

14 · 对本项目（kas-radxa-dragon）的映射总结

  • machine 链：板级 conf require qcom-qcs6490.inc（→ qcom-base.inc）；qcs6490-radxa-dragon-q6a 复用 SoC inc。
  • 核心可调点：SOC_FAMILY="qcm6490"、DEFAULTTUNE="armv8-2a-crypto"、QCOM_SELECTED_BSP(custom/base)； custom 才有 Adreno GPU、custom 6.6 内核(CodeLinaro kernel.qclinux.1.0.r1-rel)、DTBO。
  • distro/镜像：固定 qcom-wayland(+wayland/vulkan/opengl)，默认 qcom-multimedia-image。
  • cmdline：改 KERNEL_CMDLINE_EXTRA（SoC inc）；earlycon 仅 DEBUG_BUILD=1 默认有。
  • 固件/引导：packagegroup-firmware-qcm6490；systemd-boot+UKI、bootbins、partconf、ext4-only、0x80000000/4KB。
  • 刷写：firehose 三件套(prog_firehose_ddr.elf + rawprogram*.xml + patch*.xml)； 写全引导固件须覆盖 LUN1/2(xbl) + LUN4(aop)，对应 edl-ng 写全 LUN0–5。
  • kas 等价：kas-radxa-q6a.yml 替代 repo manifest + setup-environment， 定制项放 local_conf_header 或 meta-radxa-dragon。
  • 统一分支：所有层 scarthgap / qcom-linux-scarthgap，manifest qcom-6.6.90-QLI.1.5-Ver.1.1_qim-product-sdk-2.0.1。
